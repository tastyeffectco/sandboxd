# v0.4.0 preview — Snapshots / Restore / Fork (Phase 4)

**Status: preview work on branch `feat/snapshots-fork` (off `console`). NOT merged
to `console` or `main`. Not released.**

This branch adds app-scoped snapshot history, restore, and fork on top of the
public `/v1/snapshots` subsystem. The internal `/sandbox/{id}/...` zstd snapshot
subsystem is deliberately **not** exposed.

## What's implemented
- Migration `0015` adds `snapshot.source_app_id`; capture stamps it from the
  source sandbox's `app_id` so per-app history survives the (ephemeral) sandbox.
- `GET  /v1/apps/{id}/snapshots` — tenant- and app-scoped history.
- `POST /v1/apps/{id}/restore` — **replaces** the app's current sandbox: purge
  current, then clone a fresh one from the snapshot (`.git` reset to ownerless).
  Destructive; the console confirms first.
- `POST /v1/apps/{id}/fork` — creates a **new app** and spins its sandbox from
  the snapshot; the source app is untouched.
- Console: a Snapshots panel on the app-detail screen (history list, Restore with
  a confirm dialog, Fork with a name prompt).

| **Live preview after restore/fork** | ❌ **NOT verified** | — |

**Why the live spin/preview is not verified:** restore/fork create port-3000
sandboxes, which carry the `sandboxd.managed` label that the shared host's
**production Traefik** would discover. Per the release decision, end-to-end
restore/fork/preview verification is **deferred to a real, isolated v0.4.0
deploy** (its own host or an isolated Traefik), not run on the shared box. Do not
claim "preview works after restore/fork" until that deploy exercises it.
